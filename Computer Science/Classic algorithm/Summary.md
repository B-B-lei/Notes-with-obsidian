
我们考虑最经典常见的问题求解方式：建模+ 算法

# 1.建模:

首先分析问题，将复杂问题的状态用合适的数据结构保存下来，数据结构要兼顾：
1.问题的数值value,
2.数值彼此之间的连接方式
我们可以将上述数据结构抽象为：存储方式+连接方式一体化的特殊类：Container容器；这也是为什么C++如此algorithm好用的原因：它提供了算法开发中一系列配套的基础设施。

# 2.算法:

我们可以从stupid --> smart的顺序一一分析
## 2.1暴力穷举（回溯/分治）

我们不对问题实际场景具体分析，我们只将建模后的结果进行所有可行解状态的遍历，并且遍历过程中记录最小的那个solution,这就是暴力穷举。

我们在建模完成后还不能实现所有可行解的状态搜索，需要设计相应的算法流程来实现暴力穷举状态搜索，而回溯/分治就是solution；
回溯算法与分治算法本质上并非巧妙的算法，
这两者都巧妙借用了CPU架构与C/C++语法提供的递归栈结构，可以通过递归的原理轻松地遍历所有可行解空间；或者说本质上回溯算法就是基于分治算法思想的算法实现；

## 2.2 节省开销（Cache/剪纸）

在上述暴力穷举所有可行解的过程中，我们会涉及到许多重复计算/冗余的操作；我们在回溯算法的running过程中往往会对一个： 譬如A--->B---->C/D 的，C/D才产生分支的计算过程整体从A---->B开始重新计算，多次计算，冗余计算......
分析其中不必要的开销，节省不必要的计算，这是Cache/剪纸算法的核心。
那么所有解空间中有哪些是属于不必要的计算呢？哪些是可节省的开销呢？我们可以从两方面角度入手：

1.提前返回（剪枝）
根据题目立意，很多问题的最优解在某些可行解path上运行到某个位置后，可以立刻判断出它不是目标最优解:
如： 若我们始终记录一个当前最优解value,即使遍历某个可行解路径中没有遍历完成，但是若path
的开销是累积的单调上升的，那么当path上current state已经超过该当前最优value,也就没必要继续search该branch/path了，可提前exit该path的search; 这就是“剪枝”的原理；

2.缓存共用数据（Cache）
许多不同可行解路径会共用一部分path的内容，简单的暴力穷举会反复重新compute该部分share space的过程；
我们可以类比设计系统中对于反复使用的space设立cache/buffer缓存的机制;面对compute冗余的部分，我们可以用用memory将compute结果storage下来，这样就可以避免反复计算相同的内容；
动态规划算法DP就存在这样的思想；

稍微提一嘴，我们在设计DP算法时往往会维护一个DP数组，该DP数组就是我们之前为了计算最优解的运算中间过程的“缓存”。但DP除了缓存外还隐含了一些“非均匀分布策略选择”的态度，这我们后续会提。

## 2.3 态度偏好(DP，greedy)
上述2.1+2.2内容我们介绍了如何在所有解空间中search all paths以及如何节省搜索的开销，避免冗余搜索（compute）;
上述算法是没有对题目本身性质有过多要求的，适用性广泛的算法性质。但通用型的代价就是高额的运算开销，庞大的状态可行解搜索过程......而当问题本身具备一些很好的，具体的性质时，往往问题的求解过程可以简化。

回过头来分析暴力算法的本质，它就是在所有可行解空间中进行了一次遍历操作。
而当问题本身具备很好的性质时，它的可行解空间本身就具备一种：可压缩的性质。面对这种类型的问题我们可以对症下药：我们只对压缩后的解空间进行遍历操作，寻找其中的最优解。
这大大减小了算法的实际开销，我们接下来详细分析问题需要具备怎样的性质，以及我们可以将其解空间压缩成怎样的形式来求解：
### DP算法
keyword:
最优子结构、重叠子问题、无后效性、状态转移方程、边界
上面两个item都是描述可以DP求解的问题本身具备的features.

#### 1.何为最优子结构？
对于一个**求最优解**的问题 P(n)（n 表示问题规模），若满足：

> 1. 问题 P(n) 可被**分解为若干个规模更小的子问题** P(n1​),P(n2​),...,P(nk​)（ni​<n）；
> 2. 若 S(n) 是 P(n) 的**任意一个最优解**，则 S(n) 中对应子问题 P(ni​) 的部分 S(ni​)，**必然是 P(ni​) 的最优解**；
> 3. 子问题的最优解**可通过固定规则组合**，得到原问题的最优解。

则称该问题**具备最优子结构性质**。
总结起来就是：原问题可以拆分为规模更小的子问题，子问题的最优结就是原问题的最优解贡献部分，以及拆分方式透明可见。

#### 2.何为重叠子问题？
在求解最优解的过程中，子问题的结果会不止一次被复用，此时对子问题结果的cache十分重要。
若问题本身具备上述两种性质，则可以用DP算法解决。


那么DP算法如何实现？ 
#### 3.无后效性
无后效性并非原问题自带的特性，而是我们剖析定义子问题，build状态转移方程过程中，需要满足的性质。

未来只与现在有关，与过去无关，符合probability中马尔可夫链的内容，并且与状态转移方程的定义密切相关。
我们定义状态转移方程时，要保证状态变量能够完整包含所有过去的信息，不会遗漏过去信息导致状态转移方程推导未来状态时产生错误结果，以保证无后效性
（只要当前状态的value确定了，后续的问题/future 的state只与它有关了，而该状态value是如何得到，走什么paths到达.... 与未来state再无关系）。这是简化问题的关键。
#### 4.状态转移方程
这是：根据问题具备的，最优子结构性质， 将问题的子问题用明确的状态定义出来后，建立子问题的最优解状态与问题本身最优解状态的组合方式（往往是max,min(candidates)），这就是状态转移方程。注意到DP的状态转移方程往往实现的是将多个子问题的最优解进行整合组合后的更大问题最优解。
#### 5.边界，往往是DP的起始位置，从边界开始，根据定义的状态转移方程逐步求解得到问题结。


Note:
DP 求解原问题最优解，需要先求解**状态转移方程中明确依赖的、与原问题相关的子问题**的最优解 ,所以问题的求解依托于： 已有状态转移方程中子问题的最优解的 前提，我们需要先求解相关子问题最优解。而这个过程可以归纳一直到先求出边界内容。故，DP求解往往从定义边界后，自底向上的根据状态转移方程逐步迭代求解出最终问题最优解。

Summary:
从不同的路径走到一个共同状态，而后续的状态变迁都是一样的，和之前采用何种路径到这个状态没有关系，即前面的各种决策结果由这个状态表示，在考虑后半段的决策方面没有任何区别

#### Greedy算法
与DP类似，也依托于问题本身具备：最优子结构性质，可将问题拆分为多步决策的过程。
与DP不同的是，greedy往往因为问题本身更简单，子问题选择往往更少（one path only, that is greedy）,因为问题本身的特性可以将解空间压缩成 single path;我们往往都是：分布决策下，选择当前状态的greedy策略 --->下一个state, 继续greedy策略。 



## Summary:

上述算法实现中，从stupid ---> smart,依次代表了问题本身的复杂程度（high -----> low）;当问题满足的性质越多（最优子结构是重点），那么我们可以采用专门的优化策略，否则只能用high cost的暴力穷举。
