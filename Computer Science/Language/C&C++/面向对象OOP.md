1.类与对象的编程思想可以看作一种数据结构的延申；传统数据结构需要考虑的是不同数据项之间的关联方式以及实际数据存储的方式；而类与对象则更加高层且高度封装，基本不涉及底层存储原理，但涉及到更多的内部数据结构的操作方式(方法)。
   

2.****类描述了一种数据类型的全部属性(包括可使用它执行的操作)，对象是根据这些描述创建的实体。****
   

3.就像函数可以由用户自行定义与使用函数库，类也可以通过用户定义或者使用类库(istream,ostream就是类库，其中包含的cin与cout为类实例化对象)。
 
4.用户或者使用类的应用程序，不能直接访问类或对象实际内部所有内容(可以通过定义private权限来实现这一功能)，它们只能通过类向外部提供的接口去访问类并且与类交互(通过public权限定义的类内部函数方法)；而所谓的”类向外部提供的接口“ 本质上就是类编写的时候程序员设计的类内的方法函数。￼  
实际开发一个项目的时候，设计到类的设计时，一般都将类的定义，(类中定义的数据类型数据以及类内部的方法函数的声明)放在头文件中，而具体的类中方法函数的定义实现在源.cpp文件中。
 
5.类中成员变量的默认权限应为private，防止外部或者公共程序随意修改类内部数据；而访问或者执行类内部数据的操作只允许通过类中定义的public方法来实现，这就限定了类中成员的访问情况以及执行的具体操作内容(只能以public定义的方法规定的方式来访问数据并修改数据)，保证了类的成员变量的访问方式可控并且安全性，防止外界瞎改数据。
 
6.类声明中直接定义的方法一般会默认转为inline修饰的内联函数;作用等同于在类声明中声明一个函数原型(一般该声明在头文件.h中)，而在类声明外部重新用inline 修饰符实现一个对应的函数的定义。（一般在源.cpp文件中实现，并且函数名要结合类名称，inline type classname::funcname(parameter)）这样。
      

7.类的初始化函数与其它函数的声明方式都不同：类的初始化函数无任何返回值类型(因为它本身不该返回任何值，它只是对类内成员进行初始化赋值操作)；￼定义初始化函数的方法为：Classname(…){…} 通常也采用声明放在头文件，定义放在.cpp文件的方式，同时.cpp引用指定头文件，并且应用：Classname::Classname(){}这种格式来实现；￼初始化函数可以无输入参数（不建议这样做，调用这种函数说实话也没调用，但是可能存在需要的场景，比如不确定初始状态？），也可有输入参数；当有输入参数时，可通过在初始化函数声明头文件中添加函数输入参数默认值的方式实现函数重载(后续实际调用该函数的时候输入参数的数量可以更加灵活，从左到右如有未输入元素则用默认值占位)。！！！注意两点：￼(1.)带默认值参数时，仅在声明部分写出默认参数即可，后续的cpp文件中实际函数定义部分不要再写默认值赋值了，否则会导致重复定义编译器不通过。  
(2.)对于函数声明中的默认值赋值，保证从右到左的顺序赋值；实际调用函数传参时默认从左到右的顺序赋值，只要保证与声明中的默认值赋值互补或交集即合法，编译器能够识别并正确处理；(否则如果有缝编译会出错)。  
故当尝试使用无参数的默认初始化方法初始化对象的时候：￼(2.1)若类中一个构造函数(Classname())都没有,编译器会设置一个无函数体的默认初始化构造函数；  
(2.2)自己手动创建一个类中无参数构造函数,可以在该构造函数内部写一些实际的赋值内容；  
(2.3)手写一个包含所有参数的构造函数，并且每个参数都给上默认值，如此一来外部调用看似无参数的默认构造函数时，实际函数会重载为全为该默认值的有参数构造函数；但是要万分注意一点，2.3与2.2中的情况只能同时存在一个！！！要么一个完全无参数，要么一个全有默认参数，否则实际外部调用的时候编译器该执行哪一个函数呢？这两个都合法啊？故编译器遇到二义性，会报错！ 但是可以设置一个全无参数的与一个从右到左不全默认值的两个构造函数，编译器可以区分出。
      

8.CPP中的对象当想要实现对象中定义的非静态成员数据(所谓非静态指的是定义的时候未加static修饰的，属于每个对象独有的成员而不是属于该类的对象们共有的)不可修改，则需要定义const对象，该种对象一旦初始化完毕便不可修改；￼const Classname class_variable 该const对象不能使用常规的内部函数了！(因为寻常类内部函数具备直接访问类中成员的能力，并且不能保证不修改内部数据)，而是只能使用类中定义的const成员函数：￼cosnt成员函数声明方式： return_type func(…) const;￼实现方式: return_type classname::func(…) const{};￼const限定符写在函数头最后末尾部分，如此一来该函数可为const对象访问；并且当const成员函数尝试修改对象内部数据时会产生编译错误。
 
9.析构函数是类中的与构造函数对应的”终止函数“。类实例化为对象后(数据类型的实体)在内存中便有了生命周期；￼程序中定义的数据实体在内存中的生命周期取决于定义方式与存储方式：￼(1.)静态存储；数据在函数外部定义初始化，在整个程序的生命周期中都存在。  
(2.)自动存储；在函数内部定义的数据或是在作用域下({}内)，当函数运行结束或退出作用域时数据生命周期结束，由编译器自动管理(就是栈管理出栈)。  
(3.)动态存储；通过new ,delete命令创建的堆空间上的内存，在整个程序运行期间存在，只有程序员手动显式运行delete回收内存结束声明周期。￼对象也遵循上述三种不同存储方式的声明周期(对象中的成员变量)。所谓析构函数并不需要程序中显式调用，而是由编译器自动使用，当对应的对象结束其声明周期编译器会自动调用该对象所属类对应的析构函数。
   

10.CPP中OOP部分的this指针；￼this一般在定义在类中的方法函数中使用，this指针为指向该类实例化对象类型的指针，可通过this-\>member的方式来访问该对象的内部元素数据；*this返回的数据类型即为该指针指向内容即为该对象，(*this为类内部方法提供了直接访问该类实例化对象的方法，否则无法访问自己的对象数据，(原本类内部方法只能直接访问对象中元素))。
    
11.警告！！：定义函数的返回值类型时不要返回指向局部变量或者临时对象的引用或指针；当函数执行完毕后，局部变量和临时对象分配空间会被回收，数据会消失，而引用或指针将指向不存在的对象(后续视为合法空间的空间)导致程序出现错误。
    
12.CPP中的运算符重载本质上就是一种函数重载，根据运算符重载方法中的输入参数列表产生唯一的标识来选择实际用哪种方法(不允许二义性)；￼在调用运算符重载方法的时候，以+举例，+左边的对象为调用该重载方法的对象，右侧的对象为方法的传入参数对象，即：￼A+B 等价于 A.operator+(B) ￼￼注：为了确保编译器能够正确识别并处理不同运算符重载，使用的时候需要有一些限制：￼ 重载运算符必须至少一个参数是用户自定义类型(全是基本类型可能出现二义性)；  
重载运算符不能违反原来句法的规则；  
重载运算符不能更改优先级(即使重载了编译器仍然按照原先优先级处理)；  
不能创造新的运算符(与第一个类似);  
并非所有运算符都可重载(到时候使用的时候再说)。  
重载运算符的时候要考虑可解释性与理解性，不要无脑进行运算符重载。￼￼！！！有一个重要的点，对于类内部定义的运算符重载，调用对象在符号左边，传入参数在符号右边，当需求不对称或者参数类型与调用对象类型不对称的时候，该重载运算符是不具有交换性的！！！￼ 想实现实际的具有交换性的重载运算符，需使用类外定义的重载运算符（结合友元函数）。
    
13.友元函数的声明一般在需要给予该友元函数访问类内部私密数据权限的类内部public下声明； friend return_type function(classtype) 通过这种方式在类中声明，该function具有访问输入对象的内部私密数据的访问权限(通行证)；￼在外部实现友元函数的定义时，需要注意不要在函数名前加上类名与::解析附(友元函数属于全局函数，并不是类内部函数，之所以声明在类内部只是给予它通行证权限而已)。故实际调用函数并不是obj.function,而是function(obj)的方式来调用。￼￼实际有太多方式可以代替友元函数的功能，且提供如此权限并不适合OOP方式，但仍有其作用。一般结合友元函数与运算符重载功能，比如:￼在类中声明 friend return_type operator+(int m, class& A);￼如果寻常如12.中在类内部声明的成员函数作为运算重载符(以二元运算符举例)，函数的输入一般仅为另一个参数，而当参数类型不同时，显然该重载的运算不具有交换性(只能该对象位于运算符左边，输入参数在右边，位置不能颠倒)，而类内部定义运算符重载函数的另一个重载版本也显然不现实(因为类内部成员默认输入this指针指向调用对象作为参数，并且调用类内部方法只能通过obj.func)；若想运算具备交换性的特点，可以结合友元函数来实现；  
即在类外部实现一个能够访问传入对象的私密数据的函数，并也使用运算符重载功能， 类内部声明friend return_type operator+(int , class), 实现内容为： return_type operator+(int, class){} (！！注意这里实现的函数为一个全局的运算符重载函数，并不是类内部成员，不要类名解析字段！) ，后续使用的时候就可以实现当 class+int 采用类内部的operator+()重载版本，而int+class 采用类外部的operator+()版本(当然这个过程是编译器自动识别的)，从而实现运算符重载的运算交换性。
   

总结： 友元函数为全局函数，通过在类内部声明时最前面使用friend关键字声明该全局函数具备访问该类对应的对象内部私密成员的权限；￼ 友元函数通常与运算重载符函数结合使用。  
实现不可逆运算符的重载也可以通过常规外部声明运算符重载函数实现：如： type operator+(int,class){ return class+int ; } 通过在外部函数中调用类内部重载函数 来实现一致的输出，从而实现交换性！！！  
 
14.同一个类的对象中public下的成员函数可以访问同类之间其它对象的私有成员（即可以类的成员函数定义时，输入参数类型若为该类的类型(假设变量名为obj)，则函数体值可以直接使用obj.member的方式访问obj的私有成员）。
   

15.当需要标识一种数据或信息，但表示该数据的方法不唯一且等价时(好比二维空间中的向量坐标，可由直角坐标系或极坐标系表示)，类可以实现两种方式的兼容以及自动转换：￼ 可以在类成员设置时将不同方式所需要的必要信息全部作为类成员，并且设置类成员函数以实现多种表现方式的自动转换(可以通过设置private私有转化函数，并且将该方法放在类的初始化构造函数中，当输入参数为其中一种表现方式时自动调用该方法为其它表现方法的参数(此时作为类的其它成员变量)相应的赋值)。
   

16.默认情况下，当类中存在仅有一个输入参数的构造函数时(或者构造函数中有默认参数，仅有一个需要输入的参数或可输入一个参数)，程序中创建该类的对象并通过一个赋值语句可以实现该对象的创建与赋值操作，这种操作也称为类的隐式转换(编译器能够识别)。 具体过程是编译器识别该类存在输入一种参数便可以构造对象的构造函数，故编译器会根据变量作为构造函数的输入参数先创建一个临时对象；之后将该临时变量的各个成员变量复制给=左边的对象，完成赋值操作。  
该种隐式自动转换功能并非总是合适的，可在允许单个输入参数的构造函数前加上explicit关键字禁止编译器执行该类的自动隐式转换功能；但是仍允许强制转换。如classname obj; 或 ￼ obj=classname(18) obj=(classname) 18￼（假设classname类有一个输入类型为int型的单参数构造函数）￼注意这种数据转换特性基于只接收一个输入参数的构造函数，才触发。  
最后将类的这种转换特性理解为类似CPP内置数据类型，泛化理解内容。
 
还有一个重载CPP中强制转换方法的转换函数，如：operator double();这是该转换函数的声明方式。注意它必须是类内部成员函数；注意该重载函数无需显式定义返回类型，因为它转换类型即为返回类型；注意该重载函数必须无输入参数(因为实际使用是将该类的对象类型转换)。   总结：在使用隐式转换函数的时候要谨慎，推荐最好使用显式地转换函数，可通过explicit关键字限制转换函数(或一个参数的初始化转换函数)或者定义一个常规的成员函数来返回常规数值。
 
注：即使类的构造函数中不涉及动态分配内存的代码，但若使用类的过程中（如类的其它成员函数涉及到动态分配内存），则仍需设置合适的析构函数来防止内存泄露：比如队列Queue类，初始化为空队列不需要分配空间（仅需要设置一些指针指向位置，队列长度等初始变量），而Enqueue，Dequeue等方法则涉及到动态使用new delete分配内存，若不显式设置Queue::~Queue(){}析构函数，存在内存泄露的可能。
       
17.CPP编译器默认会为类提供默认的拷贝函数(浅拷贝)与默认的=运算符重载函数(执行默认的浅拷贝函数)，故Classname obj1=obj2; 本质上会执行浅拷贝函数让obj2的内容被复制到obj1中。
 
类的拷贝函数本质也是一种类的构造函数，只不过是输入参数类型为该类的一种重载形式。
 
注意拷贝函数与运算符重载(假设参数为另一个对象)存在一些区别：由于拷贝函数是构造函数的一种重载形式，故一般用于对象初始化；而=重载函数很有可能用于已有的对象重新赋值，此时应注意被赋值对象的内存空间与赋值对象内存空间相同，故需要先释放原对象空间，再重新分配指定大小空间(假设是深拷贝方式)。
    
18.CPP对于类的构造函数提供了一种初始化列表的语法，如：￼Student::Student(int num): number(num) {} ；￼它的作用与直接在构造函数体中赋值不同：使用普通函数体中赋值方式的执行过程为：先调用构造函数创建对象实体，再运行函数体中的赋值语句，类似：int a, a=5; 而初始化列表则等价于 int a=5;初始化方式；￼￼当类中包含const限定符修饰的常量成员或被声明为引用方式(&)的类成员时，无法通过常规函数体赋值语法实现初始化赋值(比如 const int a; a=5; 或者 int& a; a=5; 这种语法是禁止的) 而此时可以通过初始化列表来完成初始化(const int=5)；￼￼该初始化的语法规则为： 在函数头的输入参数列表()与函数体{}之间，通过冒号: 加 成员名(参数值) 的方式 完成赋值， 如:number(num)，就像初始化对象的语法 ; 当涉及到多个成员变量初始化时之间用 , 分隔。
 
注：该种语法仅限于类的构造函数；  
!!!是否采用初始化参数列表的语法实际对于构造对象效率有很大差别：当类中成员包含其它定义的类对象(非基本数据类型)，在进入构造函数体之前（假如是非初始化参数列表语法，函数体内赋值），CPP会默认先为成员对象调用其默认的构造函数，之后才进入函数体进行后续操作； 而初始化参数列表的语法会跳过默认构造函数的过程直接调用拷贝函数进行对象的初始化。
    
19.派生类基于基类衍生而来，继承基类中成员数据内容，且具有自己的独立于基类的成员数据； 派生类不能直接使用基类中private下成员数据（只有基类自己类下才能使用），但可通过基类中提供的public方法间接访问；  
派生类构造函数过程流程类似嵌套函数执行流程；会先构造基类构造函数，构造完基类对象后再进行派生类构造；  
派生类析构函数执行顺序与构造相反：先析构派生类，之后对基类析构；
 
可创建基类引用或指针指向派生类类型对象，则此指针仅允许访问派生类对象中属于基类类型的数据；相反派生类引用或指针不可指向基类对象！
 
可将派生类对象作为基类拷贝构造函数参数：base_obj(extra_obj);￼原理如下： 有classname(const classname& obj) 原始基类拷贝函数，而已知19.中提到的基类引用可指向派生类的转换； 实际调用拷贝函数为：￼Base_classname(Extra_classname& )，无该拷贝函数，但已有的拷贝构造函数中参数类型指向基类的引用能够指向实际函数输入的派生类对象，故允许这种方式构造基类对象（用派生类的基类数据部分赋值给新创建基类对象！）。
    
20. CPP允许基类类型指针或引用指向其派生类对象以提供泛型操作派生类对象的方式；但如果存在派生类需要对基类中定义的成员方法重载的情况，那么利用该指针-\>function访问的方法并非预期的派生类重载版本；实现这一多态重载的方式是在后续需要在派生类中重载的方法在基类中用virtual关键字声明，这样一来即使利用基类类型指针指向对应派生类对象也能正确调用对应重载版本函数实现多态，这就是虚函数。若派生类中无显示定义的重载版本，则默认继承原基类中版本的虚函数方法。
 
21.对于基类、派生类中的构造函数与析构函数是否需要virtual声明：￼构造函数不应是虚函数；参数类型不同为静态联编。  
析构函数应是虚函数。假设派生类中新添new分配动态空间存储的成员，那么当使用基类指针如：base_class* ptr=new extra_class(initial)时，后续需要回收该对象时使用 delete ptr;若基类函数定义的析构函数非虚函数，则该语句会默认运行基类版本的析构方法，仅回收基类中成员内容，而实际派生类并未回收干净；故需要使用virtual定义对应的不同重载版本；
   

注！！！： 虚函数都是与基类指针指向派生类对象联合使用的（联动）；  
普通的函数重载方案(模板、参数列表不同)都是静态联编(编译时确定)方案；  
而虚函数(virtual声明需要重载方法以及基类指针、引用指向派生类对象)是动态联编(运行时确定)方案。
   

22.派生类对于基类中的虚函数涉及到重写与隐藏两种方式：￼重写（也叫覆盖）要求函数与基类中virtual定义同名函数输入参数列表完全一致； 其余情况为隐藏，所谓隐藏不能通过基类指针实现各种派生类的重载，只能直接使用派生类类型对象或者派生类指针，并且使用的函数也是对应派生类中定义的函数版本，原基类函数被隐藏。
 
当基类中同名虚函数涉及到基类内部的多种不同参数列表的重载时，需要派生类分别对不同重载版本(参数列表)实现覆盖实现，否则假如存在未覆盖的函数，则该函数将被隐藏，派生类对象将无法使用这些未覆盖函数(因为存在其它同名方法了)。即使派生类不需要新的功能，也要定义该方法，只需要直接调用基类版本即可(假设方法权限为public)。
   

23.派生类运算符重载相关：￼当实现派生类对于基类中实现的运算符重载的拓展时，对于派生类拓展的版本输入参数为派生类对象，故不能直接利用基类指针或引用的特性来隐式转换切片(基类指针只操作识别传入的派生类对象的基类部分)，因为这样会导致函数方法陷入无穷的迭代过程： 看似隐式转换可行，但实际函数参数列表与名称(运算符重载方法)完全一致，导致函数产生递归；￼  
当拓展内容为基类内部public定义的方法时，(以=举例)显式调用 base_class::operator=(Derived_class) 这种方式来实现基类部分操作，再自行完成派生类新成员部分操作；
 
当拓展内容为friend友元函数的方式定义的运算符重载时，同理，也会陷入递归，因为友元函数是外部函数，使用方法直接为利用运算符，无类域名解析符::可用，故这里需要通过显式引用/指针类型转换来完成基类部分操作，如： 假设原方法为： ostream& operator\<\<(ostream& ,base_class )  
那么拓展外部友元函数头： ostream& operator\<\<(ostream& os, Derived& obj)  
{ os\<\<(base_class&)obj ; 显式将obj引用类型转换为基类类型完成基类\<\<友元函数重载的使用后，再进行新成员数据的逻辑操作；否则如上述会造成函数递归。
   

24.基类构造函数不为虚函数，但析构函数一般要求为虚函数(当使用基类指针访问派生类时防止通过该基类指针调用析构函数时，仅唤醒基类的析构函数，漏掉派生类成员内存空间回收导致内存泄露。）  
派生类构造函数一般在初始化参数列表中:后使用基类构造函数先对基类部分数据初始化，后续部分参数初始化列表或函数体中都可行。  
派生类对象的内存回收过程为：先调用派生类析构函数，再调用基类析构函数，且该调用过程有CPP编译器自动实现，故不推荐在派生类析构函数定义中显式调用基类析构函数；否则当涉及到new，delete分配空间时，涉及到对同一空间的两次delete操作，程序会崩溃；只需要在派生类析构中分配管理其额外的成员数据即可。
         
 
之所以有些场景选用OOP而不是面向过程的模式编程，主要是为了适应复杂的场景比如系统管理场景，以及增强代码、程序的健壮性；防止当出现需求变更时面向过程编程中的代码修改的困难与折磨(OOP的相对独立与封装的特点就很好的实现了模块的独立、解耦以及提升了代码的可维护性)。
